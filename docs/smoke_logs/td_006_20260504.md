### Splitter root-cause trace

`src/models/database.py:_split_sql_statements()` line 297 (pre-fix):
```python
if stripped.upper().startswith("CREATE TRIGGER"):
    in_trigger = True
```

This sets `in_trigger = True` *unconditionally* on any `CREATE TRIGGER` line, including the one at line 207 of `schema_postgres.sql` which is **inside** a `DO $$ ... $$;` block. When the dollar-quote closes at line 213 (`$$;`), the splitter checks `in_trigger` first and waits for an `END` line that never comes — so it eats lines 214–315 (CREATE INDEX × 15 + second function + second DO + 2 INSERTs) into a single oversized statement.

## Masked-bugs cascade (discovered post-fix)

After the schema-loader fix landed and `init_db()` started succeeding, the integration suite walked further into application code and surfaced three pre-existing bugs that had been completely invisible behind TD-006:

1. **TD-007** — datetime ISO strings rejected by asyncpg (16+ tests across 4 files).
2. **TD-008** — `round(double precision, integer)` not supported on Postgres (4 pipeline tests).
3. **TD-009** — SERIAL sequence not advanced after explicit-id fixture INSERTs (regime fixture; audit pending).

There was also an intermediate masked bug in `tests/integration/conftest.py` (`drop_all_tables` rejecting `DROP TABLE IF EXISTS` without `CASCADE` on Postgres FKs). That one was fixed inside TD-006's commit because it gates *any* test from completing setup — without it, no test could verify TD-006 worked at all.

The other three bugs are gated by file-level `pytest.mark.postgres_skip_td(...)` markers (registered in `tests/integration/conftest.py`) and tracked by their respective ticket files (`docs/planning/td_007_*.md`, `td_008_*.md`, `td_009_*.md`). Framework Lessons section now describes the cascade pattern + Mechanism F + the marker convention.

## Verdict

✅ **TD-006 is empirically fixed.** The schema loader now correctly splits `schema_postgres.sql` into 37 statements (was 18) and executes each via `exec_driver_sql`, which routes through asyncpg's simple-query protocol (multi-command-safe). `init_db()` succeeds against PostgreSQL locally and the new positive smoke test confirms the trigger function + trigger are registered.

⚠ **Full PostgreSQL test parity is NOT achieved by this commit alone.** TD-007/008/009 must close before all of `tests/integration/` runs unmasked on Postgres. CI's `db-tests-postgres` job goes from "all-tests-error-at-setup" to "everything-runs-or-skips-cleanly-with-zero-failures" — a fundamentally different shape.
